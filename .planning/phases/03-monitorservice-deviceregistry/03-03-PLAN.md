<objective>
Build `HotPlugWatcher` — the Qt main-thread component that detects USB dongle plug/unplug events.
It creates a hidden `QWidget` purely to own a Win32 HWND, registers for device-change
notifications via `RegisterDeviceNotificationW`, intercepts `WM_DEVICECHANGE` in `nativeEvent`,
debounces the 5-6 duplicate events into a single rescan, and calls `MonitorService.rescan()` to
post a discovery job onto the asyncio background loop.

Purpose: Satisfy the hot-plug halves of HID-03 (plug → discover within 1s with 500ms debounce)
and HID-04 (unplug → device goes OFFLINE on the next poll/rescan). The hidden widget is the
Phase 3 placeholder HWND owner; Phase 4 replaces it with the real application window (D-07).
Output: `src/monitor/hotplug.py` and `tests/test_hotplug.py`.
</objective>

From src/monitor/service.py (created in 03-02):
- `MonitorService.rescan() -> concurrent.futures.Future` — thread-safe; schedules discover() on the asyncio loop via run_coroutine_threadsafe. HotPlugWatcher's debounced callback calls this.

Win32 / PySide6 reference (from CONTEXT key_context):
- `PySide6.QtWidgets.QWidget` — subclass; never call .show(); use `int(self.winId())` for the HWND
- Override `nativeEvent(self, eventType, message)` to intercept Win32 messages; eventType is b"windows_generic_MSG"; decode message to a MSG struct via `ctypes.wintypes.MSG.from_address(int(message))`
- `WM_DEVICECHANGE = 0x0219`; `DBT_DEVNODES_CHANGED = 0x0007` (wParam value to act on)
- `ctypes.windll.user32.RegisterDeviceNotificationW(hwnd, NotificationFilter, Flags)`
- `DEVICE_NOTIFY_WINDOW_HANDLE = 0x00000000`
- `DEV_BROADCAST_DEVICEINTERFACE` struct with `dbcc_classguid` = USB device interface class GUID `{A5DCBF10-6530-11D2-901F-00C04FB951ED}`, OR register with DEVICE_NOTIFY_ALL_INTERFACE_CLASSES (0x00000004) to skip the GUID filter and rely on DBT_DEVNODES_CHANGED
- Debounce: `loop.call_later(0.5, _rescan)` on the asyncio loop, cancel+reschedule the returned handle on each repeat event (D-08). The handle must be created via run_coroutine_threadsafe/call_soon_threadsafe since call_later is not thread-safe from the Qt thread.

    - `__init__(self, service: MonitorService, debounce_seconds: float = 0.5)`: call `super().__init__()`;
      store `self._service`, `self._debounce = debounce_seconds`, `self._pending_handle = None`
      (the asyncio TimerHandle from call_later). Do NOT call self.show() — this widget is never visible (D-06).
    - `def register(self) -> None`: obtain the HWND via `int(self.winId())`; build the
      `DEV_BROADCAST_DEVICEINTERFACE` ctypes structure (define it inline as a ctypes.Structure with
      fields dbcc_size, dbcc_devicetype=DBT_DEVTYP_DEVICEINTERFACE(0x00000005), dbcc_reserved,
      dbcc_classguid, dbcc_name); call
      `ctypes.windll.user32.RegisterDeviceNotificationW(hwnd, byref(filter), DEVICE_NOTIFY_WINDOW_HANDLE | DEVICE_NOTIFY_ALL_INTERFACE_CLASSES)`.
      Store the returned handle in `self._notify_handle`. Raise/log on a null return (failure).
      Define all Win32 constants at module level with hex literals (WM_DEVICECHANGE=0x0219,
      DBT_DEVNODES_CHANGED=0x0007, DBT_DEVTYP_DEVICEINTERFACE=0x00000005,
      DEVICE_NOTIFY_WINDOW_HANDLE=0x00000000, DEVICE_NOTIFY_ALL_INTERFACE_CLASSES=0x00000004).
    - `def nativeEvent(self, eventType, message)`: if `eventType == b"windows_generic_MSG"`, read the
      MSG struct from the message pointer (`ctypes.wintypes.MSG.from_address(int(message))`); if
      `msg.message == WM_DEVICECHANGE` and `msg.wParam == DBT_DEVNODES_CHANGED`, call
      `self._schedule_rescan()`. Always return `super().nativeEvent(eventType, message)` (do NOT
      consume the message — return the base result so Qt keeps processing).
    - `def _schedule_rescan(self) -> None`: the debounce (D-08). Cancel any pending handle, then
      schedule a fresh one 500ms out on the asyncio loop. Because call_later is NOT thread-safe from
      the Qt thread, schedule via `self._service._loop.call_soon_threadsafe(self._arm_timer)` where
      `_arm_timer` (running on the loop) cancels `self._pending_handle` if set and assigns
      `self._pending_handle = loop.call_later(self._debounce, self._fire_rescan)`.
    - `def _fire_rescan(self)`: runs on the asyncio loop after the debounce window; clears
      `self._pending_handle`; calls `self._service.discover()` is a coroutine, so instead schedule it —
      since _fire_rescan already runs on the loop, call `asyncio.ensure_future(self._service.discover())`
      (or simply `self._service.rescan()` which is thread-safe and idempotent). Prefer calling
      `self._service.rescan()`.
    - `def unregister(self) -> None`: call `ctypes.windll.user32.UnregisterDeviceNotification(self._notify_handle)`
      if set; guard for None. Called on Phase 4 teardown.

    All Win32 interop via ctypes only (no pywin32 dependency added). This file runs on the Qt main
    thread; it must NEVER call battery_probe_chain or any hid.* directly — it only schedules
    service.rescan() onto the bg loop (CLAUDE.md invariant).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Windows OS → Qt nativeEvent | WM_DEVICECHANGE messages enter the process here; wParam/lParam are OS-controlled and must be range-checked before acting |
| Qt main thread → asyncio bg loop | _schedule_rescan crosses threads via call_soon_threadsafe; only thread-safe loop APIs are used |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-06 | Denial of Service | WM_DEVICECHANGE flood (5-6 events/plug) | mitigate | 500ms cancel+reschedule debounce (D-08) collapses the burst into a single rescan; without it the service would launch 5-6 concurrent discoveries |
| T-03-07 | Tampering | cross-thread timer scheduling from Qt thread | mitigate | Debounce timer armed via loop.call_soon_threadsafe → call_later on the asyncio loop; call_later is never invoked directly from the Qt thread |
| T-03-08 | Elevation of Privilege | RegisterDeviceNotificationW handle leak | mitigate | unregister() releases the notification handle on teardown; null-return from register is checked |
| T-03-SC | Tampering | npm/pip installs | accept | No new dependencies — ctypes (stdlib) for Win32 interop, no pywin32 added |
